# Codebase Hardening Design

## 目的

- 既存のゲーム挙動を壊さずに、保守しやすい構造へ整理する
- 将来の大きな機能追加に備えて、責務境界と失敗時の扱いを明確にする
- UI 文言の英語表現は維持する

## 対象範囲

- `pages/index.vue` の責務分割
- ゲーム状態遷移まわりの失敗ハンドリング強化
- リファクタ後の回帰を防ぐテスト補強
- 必要に応じた軽微なアクセシビリティ改善

## 非対象

- 新しいゲームモードや学習機能の追加
- 英語 UI 文言の日本語化
- データ生成フローや語彙仕様の変更
- 画面デザインの大幅変更

## 現状の問題

### 1. ページ責務の集中

`pages/index.vue` に以下が同居している。

- ページ全体レイアウト
- セッション開始前 / プレイ中 / ゲームオーバーの表示制御
- `SpeechSynthesis` 制御
- `AudioContext` による効果音制御
- `localStorage` への記録保存
- metadata 読み込み
- エラー文言整形

このため、将来の機能追加で変更点が `index.vue` に集中しやすい。

### 2. 失敗時の扱いが薄い

`answer()` や `moveToNextQuestion()` は、出題生成や状態遷移の失敗を基本的に正常系前提で扱っている。将来的に出題条件や分岐が増えると、異常系の切り分けが難しくなる。

### 3. テスト境界が粗い

主要導線のテストは存在するが、分離後に単独検証したい責務がコンポーネントに埋まっており、失敗ケースの粒度も十分ではない。

## 設計方針

### 方針 A

最重要目標は「既存のゲーム挙動を壊さずに保守しやすい構造へ分ける」こととする。

### 方針 B

将来の機能追加を見据えて拡張ポイントは用意するが、先回りした再設計は行わない。

### 方針 C

UI の見た目や基本導線は維持し、責務分離と失敗時の扱いの明確化を優先する。

## 分割設計

### `pages/index.vue` に残す責務

- ページ全体レイアウト
- `useTraditionalTrainer()` の利用
- レベル変更、回答、次問題、リセットなどのイベント接続
- UI 用 composable / component の橋渡し

### 切り出す責務

#### `composables/useTrainerAudio.ts`

- `SpeechSynthesis` の利用可否判定
- voice 選択
- 現在問題の読み上げ開始 / 停止
- pending 状態の管理

#### `composables/useFeedbackAudio.ts`

- `AudioContext` の初期化
- 正解 / 不正解効果音の再生
- ブラウザ未対応時の非致命な劣化

#### `composables/useHighScores.ts`

- `localStorage` 読み書き
- 旧形式データとの互換
- レベル別記録の更新

#### `composables/useTrainerSessionUi.ts`

- `showSessionStart`
- `isGameOver`
- `answerMessage`
- `feedbackBadge`
- そのほかゲーム状態から導出される UI 向け表示値

#### `components/SessionStartPanel.vue`

- 開始前パネルの表示責務

#### `components/GameOverPanel.vue`

- ゲームオーバー時の表示責務

必要以上に細かく分割せず、最初は責務が明確に分かれる単位に留める。

## 状態遷移とエラー処理

### 基本原則

- ゲーム進行のドメイン状態と UI 通知状態を混ぜない
- fatal と non-fatal を分ける
- 音声や保存失敗でゲーム本体を止めない

### fatal error

- 初期化不能
- 次問題生成不能
- 継続不能な出題構築失敗

これらは UI で明示的に扱い、クラッシュではなく画面状態として見せる。

### non-fatal error

- metadata 読み込み失敗
- `localStorage` 読み書き失敗
- `SpeechSynthesis` 利用不可
- 効果音 API 利用不可

これらはゲーム継続を優先し、必要最小限の表示か silent fallback とする。

### 実装方針

- `useTraditionalTrainer` に安全な遷移口を追加する
- `buildQuestion` 失敗や状態不正を UI 側で処理しやすい形に寄せる
- 文言整形は共通関数または UI composable 側に寄せる
- `index.vue` の `try/catch` は薄く保ち、失敗分類ロジックは外へ逃がす

## テスト設計

### unit

- `useTraditionalTrainer`
  - 次問題生成失敗時の扱い
  - 古い非同期結果が最新状態を壊さないこと
- `utils/trainer`
  - 出題失敗条件
  - 直近問題回避
- 新規 composable
  - 音声利用不可時の劣化
  - 記録保存失敗時の継続
  - UI 導出値の整合

### component

- 開始前表示
- 回答後表示
- ゲームオーバー表示
- ロード失敗表示
- レベル切替失敗表示
- 分離したパネルの主要イベント

### e2e

- `start -> answer -> next -> game over -> restart/reset`
- 最低限のレスポンシブ確認は維持
- 必要ならキーボード操作を追加

## アクセシビリティ改善の扱い

今回の主目的ではないが、低リスクであれば以下を同時に入れてよい。

- フォーカス表示の強化
- キーボード操作の補強
- 視覚的状態の伝達改善

ただし、英語 UI 文言は維持する。

## 実施順序

1. 既存挙動を固定するテストを先に追加または補強する
2. 音声、効果音、記録保存を composable 化する
3. UI 導出ロジックをページから切り出す
4. 開始前 / ゲームオーバー UI をコンポーネント分離する
5. 状態遷移の失敗処理を明確化する
6. unit / component / e2e を更新して回帰を防ぐ

## リスク

- 分割途中で `index.vue` と新規 composable の責務が二重化する可能性がある
- 音声制御はブラウザ依存が強く、テストダブル設計が不十分だと不安定になる
- UI 導出値を過剰抽象化すると、かえって追いづらくなる

## 成功条件

- 既存の主要導線が維持される
- `index.vue` がページオーケストレーション中心になる
- 失敗時の挙動が明示的になる
- 今後の機能追加時に責務ごとの変更箇所が見つけやすくなる
